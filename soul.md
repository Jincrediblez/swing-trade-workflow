## 一、项目定位

**Swing Trade Workflow** 是一个半自动化的美股投研工作流系统。

终极愿景：**AI Agent + 我 = 一人基金公司。**

当前阶段：AI 负责研究、数据整合、报告生成、代码维护；人负责决策与执行。

## 三、核心原则（不可违背）

### 3.1 市场范围
- **只做美股**，不覆盖 A 股、港股、加密货币等其他市场。
- 重点关注：**基本面 + 技术面 + 资金/衍生品共振**。

### 3.2 交易纪律
- **不做日内高频**（持仓周期以日/周为单位）。
- **不自动交易**——AI 只生成研究和建议，下单按钮永远在人手里。
- **不融资**（无杠杆）。
- **单只股票最大仓位 20%**；纳指 100 ETF（QQQ）和标普 500 ETF（SPY）不受仓位限制。
- **高点回撤 10% 强制休息一个季度**——风控红线，触发后停止新增仓位，只复盘。

### 3.3 数据与报告
- 所有研究报告必须可追溯：注明数据来源、分析日期、假设前提。
- 报告默认输出格式：Markdown（便于版本控制），可额外导出 PDF / Word。
- 图表必须自动化生成，禁止手动画图后人工贴图。

## 四、投研工作流

### 4.1 标准流程
美股专属工作流定义在 `WORKFLOW_US.md`，包含以下模块：

1. **宏观方向**（`templates/01_macro_direction.md`）
2. **股票筛选**（`templates/02_stock_screening.md`）
3. **基本面分析**（`templates/03_fundamental_analysis.md`）
4. **技术面分析**（`templates/04_technical_analysis.md`）
5. **资金与情绪**（`templates/05_capital_sentiment.md`）
6. **交易计划**（`templates/06_trade_plan.md`）
7. **风险管理**（`templates/07_risk_management.md`）

### 4.2 一键指令
- 常用提示词和快捷指令保存在 `scripts/us_workflow_prompts.md`。
- 新增重复性任务时，应优先将其固化为脚本或模板，而非每次都手写提示词。

### 4.3 报告输出规范
- 个股报告命名格式：`US_{TICKER}_{YYYYMMDD}.{md/pdf/docx}`
- 报告存放路径：`reports/`（源文件 + 导出文件）
- 价格/统计图表命名：`US_{TICKER}_price_chart.png`、`US_{TICKER}_stats_chart.png`

## 五、代码与文件规范

### 5.1 技术栈
- **语言**：Python 3.10+
- **依赖管理**：`requirements.txt` 是单一依赖来源，新增库必须记录于此。
- **图表**：`matplotlib` / `plotly`（优先保持一致性，当前以 matplotlib 为主）。
- **报告导出**：`python-docx`（Word）、`markdown` + 外部工具（PDF）。

### 5.2 目录结构约定
```
swing-trade-workflow/
├── scripts/          # 生产脚本（可执行）
├── templates/        # 分析模板（不可执行，纯 Markdown）
├── templates/us/     # 美股专属模板扩展
├── reports/          # 输出报告 + 图表（可清理历史，但最近 3 次保留）
├── output/           # 临时/测试输出
├── tmp/              # 临时文件（可任意清理）
├── notes/            # 想法、会议记录、非结构化思考
└── examples/         # 示例报告（供新人/未来自己参考）
```

### 5.3 Git 规范
- **主分支**：`main`，始终保持可运行状态。
- **提交信息**：使用中文，格式 `[模块] 动作: 描述`。例如：
  - `[报告] 新增: US_TSLA_20260502 分析报告`
  - `[脚本] 修复: chart_generator 中 Y 轴标签错位`
  - `[模板] 更新: 06_trade_plan 增加止损计算表`
- **敏感信息**：API Key、账户信息绝不提交到 Git，使用 `.env` 或环境变量。

## 六、AI 协作风格指南

### 6.1 交互原则
- **Kimi（本地）**：适合探索、讨论、快速迭代。可以接受模糊指令，边做边澄清。
- **Codex（云端）**：适合明确任务。下达任务时必须包含：输入、输出格式、验收标准。
- **两者共通**：AI 不得替用户做交易决策；所有涉及买卖的建议必须以"建议/分析"呈现，不能以"指令/决定"呈现。

### 6.2 代码生成要求
- 优先复用现有脚本，避免重复造轮子。
- 新增脚本必须包含模块级 docstring，说明：功能、输入、输出、依赖。
- 配置集中管理：`scripts/config.py` 是全局配置的单一入口。
- 错误处理：所有数据获取脚本必须处理网络超时和 API 限流，失败时给出可读的日志信息。

### 6.3 报告生成要求
- 使用模板填充时，保留模板原有的章节结构，不随意增删主标题。
- 数据必须标注时效性（"截至 2026-05-02 收盘"）。
- 涉及预测的部分，必须明确标注为"预测/假设"，并说明依据。
- 最终报告必须包含免责声明：本报告仅供研究参考，不构成投资建议。

## 七、质量与迭代

### 7.1 完成标准（Definition of Done）
- [ ] 代码可运行，无阻塞性错误
- [ ] 新增/修改的功能有对应输出示例（报告或图表）
- [ ] 相关模板/文档已同步更新
- [ ] `soul.md` 本身若被挑战，需在此文件更新后同步到所有 Agent

### 7.2 持续改进
- 每季度回顾一次 `soul.md`，检查原则是否仍然适用。
- 工作流中的摩擦点（"每次都要手动改 X"）应优先被脚本化或模板化。
- 新发现的数据源或工具，先在小样本验证，再纳入标准工作流。
